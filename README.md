# SDLC Agent Framework

A markdown-only orchestration system that drives full software development workflows using AI agents. No code, no dependencies, no build step — just markdown files that AI harnesses (Claude Code, Codex, or any CLI-based LLM tool) read and execute.

You describe what you want to build. The framework classifies your intent, picks the right workflow, and executes it step by step — producing requirements, architecture, working code, tests, reviews, and documentation.

## How It Works

```
You: "Build me a URL shortener API in Python with Flask"
                        |
                  [Orchestrator]
                        |
          classifies intent → greenfield
                        |
        loads workflows/greenfield.md (7 steps)
                        |
    ┌───────────────────┼───────────────────┐
    v                   v                   v
 PRD Creator      Architect          Implementer
 (requirements)   (design)           (code)
    |                   |                   |
    v                   v                   v
 Tester           Reviewer           Documenter
 (tests)          (review)           (docs + knowledge)
                        |
                  Retrospective
              (self-improvement)
```

Each agent is defined in a markdown file with purpose, inputs, outputs, numbered instructions, and quality criteria. The orchestrator reads these files, adopts each role in sequence, produces real artifacts, and tracks state throughout.

## Quick Start

### 1. Initialize a project

```bash
# Clone the framework
git clone https://github.com/sorinboia/agentic_engineering-.git sdlc-framework

# Initialize your app project
./sdlc-framework/init.sh ~/my-app
```

This creates a `.sdlc/` directory inside your project with:
- `framework_link.md` — points back to the framework
- `knowledge/` — shared knowledge base (persists across runs)
- `runs/` — per-run artifacts, state, and telemetry

### 2. Run a workflow

Open your project directory in Claude Code (or any compatible harness) and give it a request:

```
cd ~/my-app
claude
```

Then tell it what to do:

```
Build a REST API for a task manager with Node.js and Express.
It should support creating, listing, completing, and deleting tasks.
Use SQLite for storage.
```

The harness reads `CLAUDE.md` (generated by init.sh), bootstraps the orchestrator, classifies your intent as `greenfield`, and executes the full workflow.

### 3. Review checkpoints

By default, the framework pauses at checkpoint steps (PRD and Architecture) for your review. You can:
- **Approve** — continue to the next step
- **Edit** — modify the output, then continue
- **Reject** — re-run the step with your feedback

## Workflows

### Greenfield — Build from scratch

```
"Create a CLI tool that organizes files by type"
"Build a blog platform with user authentication"
"Make me a real-time chat app with WebSockets"
```

**Steps:** PRD → Architecture → Implementation (+ Test Plan in parallel) → Review → Testing → Documentation → Retrospective

### Feature — Add to existing code

```
"Add a priority system to the todo app"
"Implement dark mode for the frontend"
"Add CSV export to the reports endpoint"
```

**Steps:** Feature Spec → Design Review → Implementation (+ Test Plan) → Review → Testing → Documentation → Retrospective

### Bugfix — Diagnose and fix

```
"URLs without https:// crash the redirect"
"The search endpoint returns duplicates when paginating"
"Login fails silently when the session expires"
```

**Steps:** Bug Analysis (read-only) → Implement Fix → Verify Fix → Update Documentation → Retrospective

### Refactor — Restructure without changing behavior

```
"Extract the markdown processing into a plugin architecture"
"Split the monolithic route handler into separate controllers"
"Reduce duplication in the validation layer"
```

**Steps:** Analysis & Plan (read-only) → Plan Review → Execute Refactoring → Verify No Regressions → Update Documentation → Retrospective

### Configure — Set up project conventions and agent overrides

```
"Configure my project for TypeScript with Jest"
"Set up coding standards for this project"
"Customize the framework agents for my team's conventions"
```

**Steps:** Configure Project (interview) → Validate Configuration

### Import — Onboard an existing project

```
"Import the project at /path/to/my-app"
"Adopt the codebase at ../legacy-api into the framework"
"Bring in the existing project so I can continue with the framework"
```

**Steps:** Discovery (scan source) → Code Import (copy source into project) → Analysis (extract architecture, requirements, conventions) → Knowledge Population (write to knowledge base) → Retrospective

Every workflow run creates:

```

├── tests/                        # Working tests
├── .sdlc/
│   ├── framework_link.md         # Points to the framework
│   ├── runs/
│   │   ├── index.md              # Run history table
│   │   └── 20260428T103000Z/     # One directory per run
│   │       ├── request.md        # Your original request (verbatim)
│   │       ├── state.json        # Step-by-step execution state
│   │       ├── telemetry.json    # Full run record for analysis
│   │       └── artifacts/
│   │           ├── requirements/  # PRD or feature spec
│   │           ├── design/        # Architecture + tech decisions
│   │           ├── implementation/# Progress report
│   │           ├── review/        # Code review feedback
│   │           ├── testing/       # Test plan + test results
│   │           ├── documentation/ # User guide
│   │           └── evolution/     # Improvement proposals
│   └── knowledge/                # Shared across runs (cumulative)

│       ├── architecture.md
│       ├── conventions.md
│       ├── known-issues.md
│       ├── changelog.md
│       ├── components/           # Per-component docs
│       └── product/              # Living requirements by feature
└── CLAUDE.md                     # Bootstrapping instructions for the harness
```

## Agents

| Agent | File | Role |
|---|---|---|
| PRD Creator | `agents/prd-creator.md` | Transforms your request into structured requirements |
| Architect | `agents/architect.md` | Designs the system, chooses tech stack, defines structure |
| Implementer | `agents/implementer.md` | Writes the actual code. Also runs in analysis mode for bugfix/refactor |
| Reviewer | `agents/reviewer.md` | Reviews implementation against requirements and architecture |
| Tester | `agents/tester.md` | Plans tests (mode: planning) and writes/runs them (mode: execution) |
| Documenter | `agents/documenter.md` | Writes user docs and maintains the living knowledge base |
| Retrospective | `agents/retrospective.md` | Analyzes the run and proposes framework improvements |
| Configurator | `agents/configurator.md` | Interviews you about project specifics and generates agent overrides |
| Importer | `agents/importer.md` | Analyzes an external codebase and populates the knowledge base |

## Configuration

All settings are in `config.md`. Key options:

### Autonomy levels

```
default_autonomy: checkpoint    # Pause at checkpoint steps (default)
# default_autonomy: autonomous  # Never pause, run everything
# default_autonomy: supervised  # Pause after every step
```

### Git integration

```
git_integration: true
branch_strategy: feature-branches   # Creates feature/fix/refactor branches
# branch_strategy: trunk            # Commits directly to current branch
```

### Error handling

```
max_retries_per_step: 3        # Retry a failed step up to 3 times
escalation_threshold: 3        # Escalate the whole run after 3 total failures
step_timeout: 300              # Kill a subprocess agent after 5 minutes
```

### Self-evolution

```
retrospective_after_each_run: true
deep_retrospective_every_n_runs: 5   # Deep analysis every 5th run
max_proposals_per_retrospective: 3
track_proposal_outcomes: true        # Log whether proposals were approved/rejected
```

## Project Overrides

Customize agent behavior per project without modifying the framework:

```bash
# Add project-specific instructions to the implementer
cat > ~/my-app/.sdlc/overrides/agents/implementer.md << 'EOF'
## Additional Instructions
- Always use TypeScript strict mode
- Follow the Airbnb ESLint configuration
- All API responses must include request tracing headers
EOF
```

Override files are appended to the framework's agent definitions. The framework is never modified.

## Execution Modes

### Inline (default)

The orchestrator plays all agent roles sequentially within one session. Best for interactive use.

```
claude -p "Read CLAUDE.md and follow the framework instructions. Then: build me a todo app"
```

### Subprocess

The orchestrator spawns separate CLI sessions per agent. Enables true parallel execution and different models per step.

```
# In config.md:
default_execution_mode: subprocess
```

## Knowledge Base — Cross-Run Memory

The knowledge base at `.sdlc/knowledge/` is the framework's long-term memory. It persists across runs and accumulates understanding of your project:

- **`architecture.md`** — How the system is built (updated after each run to reflect reality)
- **`conventions.md`** — Code patterns, naming rules, error handling style
- **`product/`** — Living requirements organized by feature area
- **`components/`** — Per-component docs (purpose, API, dependencies)
- **`changelog.md`** — What changed in each run

When you run a feature or bugfix workflow, every agent reads the knowledge base first. This means the implementer follows your existing patterns, the reviewer checks against your conventions, and the architect understands what's already built.

## Self-Evolution

After each workflow run, the retrospective agent:

1. Reads the run telemetry (retries, failures, user edits)
2. Identifies what went wrong and traces it to a framework file
3. Proposes precise changes (exact text diffs) to agent definitions, workflows, or config
4. Presents proposals for your approval

Every 5th run, a deeper retrospective analyzes trends across all runs and can propose structural changes (new agents, new workflow steps, reordered dependencies).

The framework improves itself through use.

## Troubleshooting

### Pointing someone to a failed run

If something goes wrong, the run directory has everything needed to diagnose:

```
.sdlc/runs/20260428T103000Z/
├── request.md        # What you asked for
├── state.json        # Which step failed and why
├── telemetry.json    # Full timeline with retries and errors
└── artifacts/        # What each agent produced
```

Share the run ID and project path. The state file shows exactly where the chain broke, and the artifacts show what each agent received and produced.

### Common issues

| Symptom | Likely cause | Fix |
|---|---|---|
| "framework_path not found" | `.sdlc/framework_link.md` has wrong path | Re-run `init.sh` or manually fix the path |
| Agent produces empty output | Missing input artifact from a prior step | Check `state.json` — a previous step may have failed silently |
| Tests fail after feature addition | Feature changed observable behavior | Existing test assertions may need updating — check the implementer's progress report |
| Orchestrator doesn't classify intent | Ambiguous request | Be explicit: "add a feature that..." or "fix the bug where..." |

```
    ├── runs/
    └── knowledge/
```

## Requirements

- An AI CLI harness that can read files and execute instructions (Claude Code, Codex CLI, or similar)
- Bash (for `init.sh`)
- Git (optional, for automatic branching and commits)

No runtime dependencies. No package manager. No build step. The framework is pure markdown consumed by AI agents.
